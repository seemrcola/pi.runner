# Pi RUNNER 产品上下文

## 产品摘要

Pi RUNNER 是面向 Pi 用户的桌面客户端。目标用户把 Pi 作为 agentic coding/runtime 工具使用，并需要一个稳定、可检查的方式来继续会话、组织项目工作，以及监控多个正在运行的会话。

本产品不是 Pi 的替代品。Pi 仍然是写入真实会话历史、执行 agent 工作的引擎。Pi RUNNER 提供围绕这个引擎的本地桌面界面：会话发现、工作区分组、live runner 管理、prompt 输入、流式渲染、工具可见性，以及安全的视图级清理。

当前产品只支持 macOS，内测交付架构为 Apple Silicon。Intel Mac、Windows 和 Linux 不属于当前 MVP 支持范围。

## 产品类型

产品。

这是一个面向任务的开发者工具。界面应安静、紧凑、可预测，并针对重复使用优化。视觉设计服务于清晰度和速度；它不应像营销页面或演示页面。

## 主要用户

- 在多个代码库中使用 Pi 的开发者。
- 希望继续或检查 Pi JSONL 会话，但不想浏览原始文件的用户。
- 同时运行多个 agent 任务，并需要清晰区分前台/后台状态的用户。
- 在更完整的产品打磨前，验证 Pi 桌面工作流的早期内部用户。

## 核心用户任务

1. 快速开始一个普通会话。
2. 开始或继续一个绑定到工作区文件夹的会话。
3. 在不同会话之间切换，同时不停止后台工作。
4. 通过命令面板快速搜索会话、工作区和常用操作。
5. 看清哪些会话处于 idle、starting、running 或 errored 状态。
6. 阅读流式 assistant 输出、thinking blocks、工具调用和图片输入快照，并在图片查看器中浏览图片列表、识别无法显示的图片。
7. 从磁盘刷新 Pi 会话历史。
8. 从桌面视图隐藏会话或工作区，同时不删除 Pi 历史。
9. 将会话导出为只包含正常对话内容的 Markdown 文件，便于阅读、分享或归档。
10. 检查本机 Pi 安装状态，读取并编辑基础模型配置，查看已发现的本地 skills。

## 产品原则

1. Pi 是真实历史的事实来源。
   Desktop 读取 Pi sessions 并展示它们；它不改写、迁移或拥有源 JSONL 文件。
   Settings 只编辑明确属于 Pi agent 配置的 `models.json` 和 `settings.json`，不改写 Pi session 历史。

2. Desktop 拥有用户可见的投影。
   分组、逻辑隐藏、占位行和展示意图存储在 desktop metadata 中，因为它们描述的是桌面视图，而不是 Pi 历史。
   用户通过 Desktop 发送的图片快照也属于展示投影：Desktop 保存图片文件和消息映射用于回看，但不改写 Pi JSONL。

3. 前台不等于进程所有权。
   切换当前会话只改变用户看到的内容。只有当历史刷新确认它指向同一个 `sessionPath` 时，Desktop 才能把该视图附着到既有 live runner；不能因为 UI 上出现相似标题就启动第二个进程。
   切换不会立即终止后台任务；工作中的 runner 永不自动回收。空闲 runner 按前台 30 分钟、后台 10 分钟保留，并最多常驻 3 个，关闭后在下一次发送消息时透明重启。
   关闭主窗口只隐藏界面，不能停止 backend 或 Pi 任务；只有明确退出应用才结束这些进程。

4. 运行时事实来自 backend。
   Renderer 不推断进程生命周期。它消费 backend runner snapshots，并只保留草稿文本、流式 buffer 等 UI 本地状态。

5. 快速操作需要可靠恢复。
   移除会话或工作区在 UI 中是乐观操作，但 backend 确认失败时必须回滚视图并显示全局通知；确认移除后 8 秒内提供“撤销”，只恢复本次移除对象及其未发送草稿，不能覆盖期间发生的其他列表变化。
   进行中的会话不能移除；工作区包含进行中任务时也不能移除。停止任务和移除投影必须保持为两个独立动作。

6. 不可用操作必须在入口处说清楚。
   Backend 断连时，所有创建会话和刷新历史入口保持一致禁用，不能先创建一个无法发送消息的本地占位会话。设置页没有可信快照时只说明连接不可用，不把未知状态展示成“Pi 未安装”、空配置或“未发现 skills”。设置草稿未保存时，刷新不能静默覆盖草稿；返回提供继续编辑、放弃修改或保存并返回，安装 Pi 前展示来源与脚本影响确认。

7. MVP 范围要守住边界。
   任何会改变会话身份、命令来源或进程生命周期的功能，都需要在实现前给出明确设计。

## 当前 MVP 范围

范围内：

- Electron 桌面外壳。
- 本地 backend WebSocket 服务。
- Pi RPC 子进程管理。
- 前后台差异化的 idle runner 超时与容量回收。
- 从 Pi JSONL 到 SQLite 的会话索引投影。
- 会话和工作区侧边栏。
- 每个会话的 runner snapshots。
- Prompt、图片输入、follow-up、steer、abort 和流式消息展示。
- 会话与工作区的逻辑隐藏。
- 面向阅读的 Markdown 会话导出。
- 移除成功/失败的全局 Sonner 通知。
- 新建但尚未发送的会话显示为临时“草稿”；导出完成后通过通知显示实际下载文件名。
- macOS 本地 Pi 设置页：Pi 安装检测、一键安装、`~/.pi/agent/settings.json` 与 `~/.pi/agent/models.json` 编辑、本地 skills 展示。
- Dock 与菜单栏常驻入口、关闭窗口继续运行，以及有任务时的显式退出保护。
- 独立桌面宠物窗口：可从主窗口或菜单栏唤起，宠物本体、气泡、关闭按钮和透明区域都可直接拖动，短点击仍可说话或隐藏；滑稽像素怪球会在敲代码、休息、思考、散步表情间随机切换并显示随机台词，只有散步状态会随机选择方向并从屏幕一边移动到另一边。

当前 MVP 范围外：

- 编辑或改写 Pi JSONL 历史。
- 通过桌面 UI 永久删除 Pi session 文件。
- 通用文件附件协议；普通文件仍应通过路径、工具调用或 Pi 自身能力处理。
- 将 Pi slash command 发现或执行作为一等 UI。
- 基于 git metadata 的 turn 变更摘要或 agent 文件变更归因。
- Clone/reload/session migration 流程。
- 云同步或账号身份。
- 完整偏好设置系统。
- 通用多窗口工作区、跨窗口会话同步或任意窗口路由。
- 桌面宠物位置持久化、runner 状态联动、多宠物、换装、音效和碰撞物理行为。

## 信息架构

应用由一个主要工作窗口和一个可选桌面宠物窗口组成。主要工作窗口包括：

- 左侧边栏：品牌、会话、工作区、工作区操作、会话操作。
- 顶部栏：新建会话、命令面板入口、桌面宠物入口、设置入口、连接与运行状态；侧边栏顶部提供全局“同步历史”操作。
- 命令面板：通过顶部栏搜索入口或 `Cmd/Ctrl+K` 搜索并执行常用操作，或快速切换到已有会话和工作区会话。
- 设置界面：从顶部栏进入，覆盖当前应用 shell，展示 Pi 安装状态、agent 设置、模型配置文件和本地 skills。
- 主消息区：时间线导航、消息流、用户图片快照与图片查看器、thinking/tool/error 渲染、模型仍在工作状态；空状态直接提供普通会话和工作区会话入口。
- 输入区：prompt 输入、图片粘贴/拖拽、send/stop、待处理 steer/follow-up 控件。
- 全局 toast 层：乐观操作的成功/失败反馈。

桌面宠物窗口是独立的轻量浮动界面，不展示会话数据、不改变 runner 状态，也不抢占主窗口输入焦点。整个窗口表面都应支持拖动，不能要求用户寻找透明空白区；拖动按钮时不触发按钮操作，短点击仍按原语义响应。敲代码、休息和思考状态保持原位；进入散步状态时随机从宠物当前所在显示器的左侧或右侧出发，抵达另一边后停止。随机表情和短台词提供陪伴感，但频率必须克制，不能干扰编码工作。

首屏就是可用应用，不是 onboarding 或营销页。空状态应让下一步操作足够明确，优先用“普通会话”“选择工作区”这类动作入口，而不是讲实现细节。

侧边栏中的会话和行内操作都必须可通过标准键盘导航访问：`Tab` / `Shift+Tab` 移动焦点，`Enter` / `Space` 选择会话或打开操作菜单；键盘焦点不能依赖 hover 才可见。

## 交互语言

优先使用用户动作语言，而不是实现语言。

推荐：

- New conversation
- Refresh history
- Remove conversation
- Remove workspace
- Follow up
- Stop
- Running

只有当受众明确偏技术，并且该术语是最清楚的表达时，才使用内部术语：

- sessionPath
- runner
- snapshot
- Pi JSONL
- SQLite projection

避免在主要 UI 文案中暴露原始实现语言：

- 当 "history" 或 "conversations" 更清楚时，不把 `sessions` 作为用户可见标签。
- 避免使用 `steer`，除非该交互已经被解释过或重命名过。
- 不在普通控件中暴露 `projection`、`lease`、`RPC`、`JSONL`。

## 可靠性预期

- 显示为 running 的会话只能接收属于自己的 backend events。
- 一个 canonical sessionPath 最多只能有一个 live writer；重复打开及软链接别名应附着到既有 runner，而不是创建第二个可发送入口。跨 backend runtime lock 无法确认 owner 时必须拒绝启动。
- 从侧边栏删除只是在 desktop projection 中隐藏该项；不会删除 Pi 历史。
- `starting`、`running`、`stopping`、`terminating` 或 `termination_failed` 会话及包含这些会话的工作区不得移除。
- 关闭主窗口后 backend 和运行任务必须继续；有任务或任务状态未知时，退出应用必须明确说明影响。
- Backend 重连后必须用完整 runner 列表收敛旧流式状态；已不存在的任务不能继续显示为运行中或接收下一轮输出。
- 停止请求只能作用于真实运行中的 runner；过期或重复操作不能制造永久 `stopping` 状态。
- Prompt 和停止只有在 Pi 明确 ACK 后才能改变 runner phase；发送到 stdin 不等于任务已接受。
- 正常退出必须等待 Pi `close`；TERM 超时后清理已确认身份的后代进程。进程或锁身份不确定时保持阻塞，不能冒险启动第二个 writer。
- Electron 异常退出后，backend 必须通过父进程 watchdog 清理自身 process group；backend 的重启与清理事实必须跨重启保留，便于定位反复崩溃。
- 同时最多运行 4 个 active runner；达到上限时拒绝新任务，不影响已有任务。
- 重复唤起桌面宠物必须复用同一个窗口；显示、隐藏或拖动宠物不能改变主窗口、backend 或 Pi runner 生命周期。
- 消息中的 HTTP(S) 链接必须由系统浏览器打开，应用窗口不能导航到外部页面或执行危险链接协议。
- 乐观删除必须被确认或回滚。
- 同步失败应保留现有侧边栏投影。
- JSONL 最后一条记录尚未写完整时，同步必须保留旧投影并在文件变化后重试，不能永久吞掉末尾消息。
- Backend 断连时，应清理 request bookkeeping，并停止把陈旧 runtime status 展示成实时真相。
- Backend 断连时，未完成的设置读取、保存和安装状态必须解除锁定，同时保留用户草稿供重连后重试。
- Pi 启动、stdin 写入和设置保存失败必须只反馈一次并保留可恢复草稿；并发设置响应不能互相覆盖。

## 设计调性

UI 应像一个服务于严肃编码工作的紧凑桌面工具：

- 克制的深色表面；
- 低视觉噪音；
- 小而稳定的控件；
- 清楚的状态指示；
- 可预测的菜单；
- 不使用装饰性页面 section；
- 不使用营销式卡片；
- 除状态反馈外，不使用炫技动画。
- 桌面宠物可以使用低帧率马赛克动作，但必须支持减少动态效果的系统偏好，且不能引起窗口尺寸跳动。

产品可以拥有技术身份，但当个性和清晰度冲突时，清晰度优先。

## 成功标准

当出现以下情况时，说明 MVP 正在成功：

- 用户可以开始和继续 Pi 工作，而不需要思考文件位置。
- 后台任务能始终正确关联到自己的会话。
- 侧边栏分组符合用户对 sessions 与 workspaces 的心智模型。
- 用户可以安全移除杂乱内容，并能从 backend 失败中恢复。
- 添加新的生命周期功能时，不需要在 renderer 中复制进程状态。

## 反目标

- 不成为 `~/.pi` 的通用文件浏览器。
- 不构建第二套 chat history 事实来源。
- 不用本地-only UI 状态掩盖进程失败。
- 不在语义设计清楚之前，让 slash commands 或会改变生命周期的命令看起来已经被支持。
- 不为可逆的视图级隐藏增加沉重的确认流程。
