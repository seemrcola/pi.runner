# Pi RUNNER macOS 内测 E2E 验收清单

本文档用于每个 macOS 内测候选版本的人工黑盒验收。自动化测试负责验证模块和协议边界；本清单负责验证打包后的 App、真实 Pi 进程、文件系统和用户操作能否组成完整工作流。

## 发布门禁

- P0 项必须全部通过。
- 不得遗留 S0、S1 或 S2 问题；S3 可以记录后随版本处理。
- 每轮验收记录构建 commit、Pi 版本、macOS 版本、机器架构和测试人。
- 失败项必须附复现步骤、截图或录屏，以及相关日志。

问题级别：

- S0：造成 Pi 历史、配置或用户文件不可恢复地损坏。
- S1：应用无法启动，或新建、发送、继续会话等核心路径完全不可用。
- S2：核心能力可使用但结果错误、状态串线、无法恢复，或需要明显绕行。
- S3：文案、样式、轻微交互或不影响任务完成的问题。

### 准备工作

- [ ] 使用专用测试工作区，不在重要代码库中执行会修改文件的 prompt。
- [ ] 确认 `pi` 命令可用，且测试模型已经配置并能从终端完成一次最小调用。
- [ ] 如需测试设置保存，先备份 `~/.pi/agent/settings.json` 和 `~/.pi/agent/models.json`。
- [ ] 记录测试前 `~/.pi/agent/sessions` 中的目标测试 session，便于确认 Desktop 没有删除源历史。
- [ ] 关闭旧版 Pi RUNNER，避免不同构建同时使用同一测试环境。

建议创建一个可丢弃的工作区，其中至少包含 `README.md` 和一个可修改的文本文件。不要把仓库自己的源码目录当作写文件测试目标。

## P0 自动化基线

在候选 commit 上依次执行：

```bash
npm ci
npm run verify
npm run build
npm run package
```

- [ ] 所有命令退出码为 0。
- [ ] `npm run verify` 没有失败或跳过的测试。
- [ ] `release/` 中生成当前版本的 arm64 DMG 和 ZIP。
- [ ] 构建日志中没有新增的未解释错误；未签名和未公证是当前 macOS 内测版的已知分发限制。

## P0 安装与启动

- [ ] 从本轮生成的 DMG 安装 App，不复用旧的 `.app`。
- [ ] 首次启动能进入主界面；未签名构建按内测说明完成 macOS 手动放行。
- [ ] 主窗口不是白屏，侧边栏、顶部栏、消息区和输入区均可见。
- [ ] 顶部连接状态在合理时间内变为已连接，没有持续闪烁或反复重连。
- [ ] 已有 Pi 历史能够显示，标题、工作区归属和更新时间没有明显错误。
- [ ] 打开设置后能识别当前 `pi` 可执行文件，并能返回会话界面。
- [ ] 菜单栏显示 Pi RUNNER Template 图标，状态摘要与当前任务数量一致。
- [ ] 连续打开第二次 App 只激活已有窗口，不出现第二套窗口、backend 或持续重连。

## P0 普通会话主链路

1. 点击顶部“新会话”，选择普通会话。
2. 发送：`只回复 E2E_READY，不要调用工具。`

- [ ] 创建后立即出现本地会话视图，没有重复侧边栏条目。
- [ ] 状态按实际过程从启动中/运行中回到空闲，不会长期卡在运行中。
- [ ] 用户消息立即出现，assistant 内容流式显示且最终只出现一次。
- [ ] 最终回复包含 `E2E_READY`，没有把消息投递到其他会话。
- [ ] 切换到其他会话再切回，消息、滚动和输入草稿没有串线。

## P0 工作区与工具调用

1. 新建工作区会话并选择专用测试目录。
2. 发送：`在当前工作区创建 e2e-output.txt，内容仅为 E2E_FILE_OK，然后告诉我完成了什么。`

- [ ] 会话显示在正确的工作区分组中，工作目录与所选目录一致。
- [ ] 工具调用显示运行和完成状态，展开后能看到命令、参数、输出或 diff。
- [ ] `e2e-output.txt` 实际位于测试工作区，内容准确，没有写到其他目录。
- [ ] assistant 最终消息与实际文件结果一致。
- [ ] 置顶、折叠工作区后重启 App，视图状态仍然保留。

## P0 后台运行与会话隔离

1. 在工作区会话发送：`执行 sleep 8; echo E2E_BACKGROUND_OK，命令结束后只回复 E2E_BACKGROUND_OK。`
2. 状态变为运行中后，立即切到另一个会话并正常浏览或发送一个最小 prompt。
3. 等待后台任务结束后切回原会话。

- [ ] 切换会话不会停止后台任务。
- [ ] 侧边栏能区分前台和后台会话的运行状态。
- [ ] 后台产生的工具事件和最终回复只出现在原会话。
- [ ] 两个会话的草稿、pending 指令、流式文本和错误状态互不污染。
- [ ] 任务运行中时，会话“移除”不可用；工作区内存在运行任务时，“移除工作区”不可用。
- [ ] 关闭主窗口后菜单栏图标仍存在，后台任务继续；从 Dock 和菜单栏都能恢复窗口并看到任务完成结果。
- [ ] 运行任务时按 `Cmd+Q` 会显示退出影响；取消和隐藏窗口不会停止任务，“退出并停止任务”才真正退出。
- [ ] “退出并停止任务”后使用活动监视器或 `ps` 检查，不残留对应 backend、`pi --mode rpc` 或本轮 bash 工具进程。
- [ ] 仅修改 `models.json` 或仅修改 `settings.json` 草稿时按 `Cmd+Q`，都必须提示存在未保存设置。

## P0 追加指令与停止

### 追加指令

1. 发送一个至少运行数秒的任务。
2. 运行期间在输入区输入 `最终回复末尾追加 E2E_FOLLOW_UP` 并发送。

- [ ] 输入内容进入待处理区域，不会丢失或立即出现在错误会话。
- [ ] 可以提交或移除待处理指令。
- [ ] 当前任务自然结束并自动发送排队 follow-up 后，对应用户消息持续留在消息流中，不会被紧随其后的历史刷新隐藏或重复。
- [ ] 提交后 Pi 能接收指令，最终结果包含 `E2E_FOLLOW_UP`，或界面明确展示 Pi 返回的失败原因。

### 停止

1. 发送：`执行 sleep 30; echo SHOULD_NOT_FINISH。`
2. 状态变为运行中后点击停止按钮。

- [ ] 停止操作有即时反馈，任务不会继续显示为永久运行中。
- [ ] 在任务自然结束的同时点击停止，状态最终仍会回到空闲或错误，不会永久停在“停止中”。
- [ ] 停止后可以在同一会话再次发送 prompt。
- [ ] 停止一个会话不会影响其他正在运行的会话。

## P0 图片输入与回看

1. 准备一张内容可识别的 PNG 或 JPEG 截图。
2. 分别验证粘贴或拖拽中的至少一种输入方式。
3. 发送图片并要求 Pi 描述其中一个明确元素。

- [ ] 图片在发送前显示缩略图，可以打开预览并移除。
- [ ] 预览支持关闭；多图时可以切换并正确播报位置。
- [ ] 图片随用户消息发送，Pi 的回复能证明模型收到图片。
- [ ] 图片加载失败时显示明确失败状态，不会留下空白遮罩。
- [ ] 刷新历史并重启 App 后，已发送图片仍能在原消息中回看。

## P0 历史、导出与逻辑移除

- [ ] 点击刷新历史后，当前会话不重复、不消失，运行中的本地状态不被旧历史覆盖。
- [ ] “同步历史”入口位于侧边栏顶部应用级工具区，不属于会话或工作区分组；有任一任务运行时入口和命令面板命令均禁用，完成后显示更新数量或无新会话反馈。
- [ ] 完全退出并重启 App 后，测试会话仍可读取和继续。
- [ ] 导出的 Markdown 只包含正常对话正文，不包含 thinking、工具输出、内部 id 或协议字段。
- [ ] 移除一个测试会话后，它从 Desktop 视图消失，但对应 Pi JSONL 文件仍存在。
- [ ] 移除测试工作区后，分组从 Desktop 视图消失，但工作区文件和 Pi JSONL 历史仍存在。
- [ ] 移除确认后 8 秒内点击“撤销”，被移除会话/工作区恢复，窗口期间新增的其他会话仍保留。
- [ ] 撤销包含未发送文本或图片的本地草稿会话后，草稿内容仍完整；撤销失败显示全局通知，不写入聊天记录。
- [ ] 在任务从空闲切换为运行的竞态下尝试移除，Backend 拒绝后投影保持可见并显示“任务进行中”。

## P0 设置安全

本节会读写真实 Pi 配置。只在已经备份配置、且测试完成后能够恢复时执行。

- [ ] 设置页展示的 `settings.json`、`models.json` 路径与实际文件一致。
- [ ] 修改草稿后，“刷新”不可用，不会覆盖未保存内容。
- [ ] 修改草稿后返回会提供继续编辑、放弃修改、保存并返回；一键安装会先展示 pi.dev 来源和脚本影响确认。
- [ ] 点击“还原”后恢复最近一次 backend snapshot，保存按钮重新禁用。
- [ ] 输入无效 JSON 并保存时显示错误，磁盘上的原配置内容不变。
- [ ] 保存一份有效的测试修改后，文件保持有效 JSON，重新打开设置能读到保存结果。
- [ ] 保存 `settings.json` 不会覆盖未保存的 `models.json` 草稿，反向操作同样成立。
- [ ] 任一配置正在保存或 Pi 正在安装时，其他设置请求不可重复提交；保存期间继续编辑的草稿在回包后仍完整。
- [ ] 设置读取、保存或安装进行中断开 backend 后，进行中状态会结束，草稿仍保留，重连后可以重试。
- [ ] 断网或让 pi.dev 安装脚本下载失败时，安装结果显示失败，不会把空脚本误报为成功。
- [ ] 安装进行中退出 App 后，不残留 installer、curl 或 sh 进程；再次启动可以重新发起安装。
- [ ] Skills 列表可加载；打开文件夹动作指向正确位置，失败时有可见反馈。
- [ ] 验收结束后恢复备份配置，并再次启动 Pi 确认配置有效。

## P0 命令面板与窗口布局

- [ ] `Cmd+K` 和顶部搜索按钮都能打开命令面板，焦点自动进入搜索框。
- [ ] 可以按会话标题、工作区路径和运行状态搜索，并用键盘切换会话。
- [ ] 新建、刷新历史和设置命令复用正常主流程，没有产生额外会话或重复请求。
- [ ] 新建但未发送的会话显示“草稿”及临时性提示；重启后消失属于预期行为，发送第一条消息后重启仍可恢复。
- [ ] 导出会话后显示实际下载文件名，文件内容可正常打开。
- [ ] 点击消息中的 HTTP(S) Markdown 链接后，系统默认浏览器打开目标地址，Pi RUNNER 主窗口仍停留在当前会话且可继续操作。
- [ ] `file:`、`javascript:` 等非 HTTP(S) 链接不会执行、打开本地文件或让 Pi RUNNER 导航离开应用页面。
- [ ] 在最小支持尺寸 860x560 和常用尺寸 1440x900 下，没有控件重叠、正文遮挡或页面级横向滚动。
- [ ] 长标题、长路径、长命令和长代码行不会撑破侧边栏、顶部栏或工具块。
- [ ] 图片查看器、命令面板和新建会话弹窗打开时，Escape、焦点恢复和背景交互符合预期。

## P0 桌面宠物与多窗口

- [ ] 点击顶部宠物按钮后出现透明的黄绿色滑稽像素怪球窗口，主窗口输入焦点不被抢走。
- [ ] 再次点击按钮不会创建第二只宠物或重复窗口；隐藏后再次唤起能恢复同一个窗口。
- [ ] 菜单栏“显示桌面宠物”也能创建或恢复宠物窗口。
- [ ] 从宠物本体、气泡、关闭按钮和透明区域都可以拖动窗口；拖动按钮不会误触说话或隐藏，短点击仍正常响应。散步过程中拖动后自动移动暂停约 5 秒，窗口不能被拖出所有显示器，没有矩形背景、残影或明显裁切。
- [ ] 敲代码、休息、思考状态不移动窗口；散步状态会随机从宠物当前显示器左侧或右侧出发并走到另一边，到达后停止，拖到副屏后不会跳回主屏。
- [ ] 敲代码、休息、思考、散步四种表情都能在合理时间内出现，且视觉上可以区分。
- [ ] 随机台词不溢出窗口，气泡出现和消失不会改变窗口尺寸。
- [ ] 点击气泡中的关闭按钮会隐藏宠物；主窗口、backend 和运行中的任务不受影响。
- [ ] 关闭主窗口后宠物窗口和菜单栏生命周期符合预期；显式退出应用后所有窗口都关闭。
- [ ] 启用 macOS“减少动态效果”后，宠物不再跳动或呼吸缩放，但当前姿态仍可辨识。

## P1 恢复能力与长时间运行

这些项目不阻塞每次小版本，但发布较大内测版本时应执行。

- [ ] Pi 返回认证错误、模型不可用或图片不支持时，错误只显示在目标会话，并允许修复后重试。
- [ ] 网络短暂中断后，backend 和 renderer 能恢复连接，旧运行状态不会继续伪装成实时状态。
- [ ] 手动同步历史过程中断线后，旋转状态会结束；重连后可以再次同步。
- [ ] 连接状态依次能表达正在连接、已连接、正在重新连接和连接失败自动重试。
- [ ] Pi 可执行文件缺失时，启动只显示一次明确错误，不会先进入已启动状态或发送首条 prompt。
- [ ] Pi 退出前写入最后一批 stdout 事件时，这些事件先完成渲染，再出现 terminal runner 状态；terminal 后不会新建流式消息。
- [ ] Pi JSONL 最后一行临时截断时保留已有历史；文件补全后再次同步能恢复末尾消息。
- [ ] 在 assistant 流式输出期间意外结束 backend 子进程后，Electron 能按策略重启并恢复连接；旧消息和未结束工具停止 streaming，不再显示“Pi 仍在工作”，随后发送的新任务使用新的 assistant turn。
- [ ] 同时运行 4 个会话后，第 5 个 active runner 被明确拒绝；已有任务不被终止，idle runner 仍按容量策略收敛。
- [ ] 在 assistant 流式输出期间强制结束 backend 后，旧 backend process group 和 Pi 工具进程先消失，随后才出现新 backend；同一 session 不产生两个 writer。
- [ ] 强制结束 Electron main 后，backend parent watchdog 能清理 backend/Pi/installer process group；再次启动应用不会长期占用旧端口或创建第二个 session writer。
- [ ] `~/pi.runner/data/runtime/process-lifecycle.jsonl` 能定位 spawn、phase、TERM/KILL 和退出结果，且不包含 prompt、模型输出或完整 session 路径。
- [ ] `~/pi.runner/data/runtime/backend-supervisor.jsonl` 能定位 backend spawn、ready、group cleanup 和 restart，且不包含 token、环境变量或 backend stdout/stderr 正文。
- [ ] 前台 idle 约 30 分钟、后台 idle 约 10 分钟后，重新发送 prompt 能透明恢复 runner。
- [ ] 连续运行 2 小时并多次切换、发送、停止和刷新后，没有明显内存持续增长或界面变慢。
