# AGENTS.md

本文件约束本仓库内 AI Agent 的默认工作方式。除非用户明确给出更高优先级的指令，否则所有代码、文档、设计建议和提交说明都应遵守这里的底线。

## 项目阶段与改动策略

- 当前项目处于 MVP 阶段，没有历史兼容负担。
- 面对不合理的旧实现时，优先做直接、清晰的结构性改动；可以接受破坏性变更，不要为了兼容旧逻辑长期叠补丁。
- 破坏性改动必须把调用方、测试、类型、文档和用户可见行为一起收口，避免只改一半。
- 如果保留旧逻辑只是为了临时过渡，需要在代码或提交说明里写清楚原因和后续删除条件。

## 设计建议

- 给出设计或架构建议时，优先参考 GitHub 上成熟项目的实践，或参考主流操作系统的交互与信息架构设计。
- 不要只给抽象审美判断；建议应说明借鉴对象、适用原因，以及对当前 MVP 的取舍。
- 如果建议来自推断而非可验证参考，需要明确标注这是推断。

## 代码与注释

- 代码应优先保持简单、直接、可删除；不要为了“未来可能需要”提前抽象。
- 重要逻辑必须写注释，尤其是：
  - 跨进程、跨模块、异步生命周期相关的边界；
  - 数据结构、协议、状态机和持久化格式；
  - 破坏性迁移、清理旧数据、删除兼容逻辑的原因；
  - 表面看起来反直觉但实际必要的实现。
- 代码中的注释、文档、说明性文本尽量使用中文；标识符、第三方 API 名称、协议字段和行业通用术语可以保留英文。
- 注释应解释“为什么这样做”和“这里的约束是什么”，不要重复代码表面含义。

## 文档同步

- 每次更改代码时，都要检查是否需要同步更新 `DESIGN.md`、`PRODUCT.md` 和 `README.md`。
- 如果仓库同时存在大小写不同的设计文档文件，需要确认项目实际使用的是哪一个，并避免无意制造重复来源。
- 涉及产品行为、交互流程、架构边界、运行方式、开发命令或用户可见能力的变更，通常都需要同步文档。
- 如果判断不需要改文档，最终说明里要简短写明“已检查，无需更新”的理由。

## 提交信息

- Git commit 信息要详细、清楚，能让后来者只看提交记录就理解改动意图。
- 推荐格式：

```text
type(scope): 简短说明

说明为什么要改、改了什么、影响范围是什么。
如果有破坏性变更，明确写出 BREAKING CHANGE。
列出已运行的关键验证命令或说明未运行原因。
```

- 不要使用含糊的提交信息，例如 `fix`、`update`、`misc`。

## 完成前检查

- 完成任务前至少检查：
  - 代码是否仍符合 MVP 阶段“直接改对”的策略；
  - 重要逻辑是否已有必要中文注释；
  - `DESIGN.md`、`PRODUCT.md`、`README.md` 是否需要同步；
  - 是否运行了与改动风险匹配的测试或验证；
  - 最终回复是否清楚说明改动、验证结果和未覆盖风险。
